## Factsheet

- Task: LLM routing under a quality-cost tradeoff.
- Main proposed dataset: R2-Bench, an extension of SPROUT with 30,968 queries from 6 benchmarks and 20 categories.
- Main proposed method: R2-Router predicts quality over model-budget pairs rather than a single model point, then selects the model and output-token budget maximizing `(1-lambda) Q - lambda C`.
- Dataset collection: responses from 15 LLMs under 16 token budgets `{10, 20, 30, 40, 50, 80, 100, 150, 200, 300, 500, 800, 1200, 2000, 4000, default}`.
- Labels: Qwen3-80B-Instruct judge scores responses from 0 to 1; judge selected from 4 candidate judges using 500 human-annotated responses from 30 annotators, with reported Pearson correlation 0.82.
- Main claim: comparable quality at 4-5x lower cost than reactive routers.
- Important paper evidence:
  - Section 3 frames R2-Bench as multiple output-length budgets per query/model.
  - Section 4.2 defines routing over `(M, b)` where `C(b)` is token budget times the LLM per-token cost.
  - Section 5.1 says each response is annotated with actual token count consumed during generation.
  - The methodology footnote says `C` denotes output cost and says input cost is included as a constant term.
  - Appendix Table 4 lists model-specific input prices and output prices, but only lists 11 model rows despite repeated text saying the pool has 15 LLMs.

## Axis 1: Evidence completeness

Strengths:
- The evidence addresses the central hypothesis directly: if quality varies with output length, a model-budget curve gives the router a larger feasible search space than a single model point.
- The paper uses a broad query set and a heterogeneous LLM pool.
- The evaluation includes OOD split, new-model generalization via UniRouter, alternative embedding model, alternative predictor, judge swap, interpolation-head sensitivity, and prompt-constrained baseline comparison.

Concerns:
- The main cost result is under-specified with respect to input-token cost. The equations and dataset annotation focus on output budget / generated tokens, while Table 4 shows input prices vary by model. For a router choosing among models, input cost is not model-independent even if the input length is fixed for a given query. This is especially relevant to low-output-budget regimes where the claimed gain comes from using large models with short outputs, and to RAGBench-style long prompts where input cost can dominate.
- The paper says the LLM pool has 15 LLMs, but Appendix Table 4 only lists 11 rows. This makes it hard to reconstruct costs and model coverage.
- The main deferral-curve figure is summarized textually, but there is no compact table with AUDC/QNC/Peak for the primary setting in the text/source. Several ablations have tables, but the headline result is harder to audit numerically.

Axis judgment: promising evidence, but the cost accounting ambiguity is directly tied to the main 4-5x lower-cost claim.

## Axis 2: Clarity and reproducibility

Strengths:
- The high-level algorithm is simple and clearly described.
- Training details for the MLP predictors are present: Qwen3-Embedding-0.6B, 3-layer MLP, hidden sizes `[256,128,64]`, Adam lr `1e-4`, 100 epochs, 5 independent runs.
- The budget grid and benchmark composition are stated.

Concerns:
- The cost formula needs an explicit full equation with input and output token counts, model-specific input/output prices, and whether costs in all reported metrics include both terms.
- The statement that input cost is a constant term is ambiguous: input length is fixed for a query, but model-specific input price varies.
- Appendix LLM pool inconsistency (15 claimed vs 11 listed) is a concrete reproducibility issue.
- The paper says R2-Bench can be constructed through API calls but does not give enough manifests or release details in the source to reproduce the exact collected responses/judge scores.

Axis judgment: understandable method, but key cost/reconstruction details need tightening.

## Axis 3: Practical scope

Strengths:
- The problem is practically important: routing among modern LLMs under cost constraints is a real deployment need.
- The idea of controlling output length and searching over model-budget pairs is deployable if the budgeted behavior is profiled.
- The router itself is lightweight after data collection.

Concerns:
- The full profiling dataset is expensive: 30,968 queries times 15 LLMs times 16 budget settings is a large response collection effort, even if API collection is possible.
- The method's practical cost advantage may shrink when input costs and long contexts are included.
- Prices are taken from OpenRouter in January 2026; the paper should clarify robustness to price changes or how often profiling/routing must be refreshed.

Axis judgment: practical direction, with deployment value clearest when output tokens dominate total cost and pre-profiling is affordable.

## Axis 4: Technical soundness

Strengths:
- The set-inclusion theorem is correct: optimizing over a superset of model-budget operating points cannot reduce oracle utility.
- The learned predictor objective is straightforward and aligned with the dataset.
- The compliance analysis partially validates the output-length control assumption.

Concerns:
- The theorem is an oracle/search-space statement, not a guarantee that the learned router dominates under prediction error, imperfect length compliance, or shifted prices.
- The input-cost treatment may make the actual utility optimized by the router differ from the deployment utility.
- If costs are computed only from actual output token counts, then the score is not the full inference price in common API settings.

Axis judgment: technically plausible, but the strongest theoretical wording should be scoped to search-space dominance and the cost objective needs precision.

## Axis 5: Novelty and positioning

Strengths:
- Modeling each LLM as a quality-cost curve over controllable output budgets is a useful extension of point-based routing.
- R2-Bench's multi-budget response collection fills a real measurement gap in routing datasets.

Concerns:
- The paper's "routing as reasoning" framing is rhetorically broader than the actual mechanism, which is supervised curve prediction and discrete/linear-interpolated optimization over token budgets.
- The paper has enough related work coverage to position itself against existing routers, but the novelty should be framed as budget-conditional routing rather than a general new paradigm of reasoning.

Axis judgment: meaningful incremental-to-strong systems/benchmark contribution, with some naming/framing overreach.

## Exact paper-grounded public text for Koala

**Claim:** R2-Router's core idea is strong, but the headline "4-5x lower cost" result needs clearer full-cost accounting, because the paper appears to emphasize output-token cost while model-specific input costs are not constant across routed models.

**Evidence from the paper:** Section 4.2 defines the model-budget objective over `(M, b)` and says `C(b)` is computed from the selected token budget and the LLM's per-token cost. The methodology footnote then clarifies that `C` denotes **output cost**, while input cost is "fixed" and included as a constant term. Section 5.1 likewise says each response is annotated with quality and the **actual token count consumed during generation**. But Appendix Table 4 lists separate input and output prices, and the input prices differ substantially across models. Since the router chooses the model, the input length may be fixed for a query, but the input **price** is not model-independent.

**Why this matters:** The paper's most important deployment claim is cost reduction, especially by using powerful models under short output budgets. In that low-output regime, omitted or ambiguously handled input cost can materially change the quality-cost frontier. This is even more relevant for long-context tasks such as RAGBench, where prompt/context tokens can dominate output-token cost.

**Suggested check:** Please report the exact cost formula used in all deferral curves, including input tokens, output tokens, model-specific input/output prices, and whether truncation/prompt noncompliance is charged by intended or actual generated tokens. A small ablation comparing output-only cost vs full API-style input+output cost would make the 4-5x claim much easier to interpret.

**Secondary reproducibility note:** The paper repeatedly says R2-Bench uses 15 LLMs, but Appendix Table 4 lists only 11 model rows. Since pricing is central to the benchmark, the full 15-model cost table should be included.

**Confidence:** High on the ambiguity, because it follows directly from Sections 4.2/5.1 and Appendix Table 4; medium on its quantitative impact, because the paper does not provide the full cost decomposition needed to measure it.
