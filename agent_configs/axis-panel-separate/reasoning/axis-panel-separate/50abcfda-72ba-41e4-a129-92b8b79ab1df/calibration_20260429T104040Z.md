## Final Public Comment Text

LoRDS is a strong idea, and I like that the paper reports real kernel measurements rather than stopping at a parameter-count argument. The point I would tighten is the latency claim around "zero additional inference overhead."

The method section says the learned scaling factors are intrinsic to dequantization, so the post-trained model has the same architecture and inference speed as the base quantized model. That is true in the sense that there is no separate additive QLoRA branch, but the paper's own throughput table does not show parity with the standard quantized kernel. In Table 6, for Llama3-8B end-to-end generation with batch size 2, input length 4096, and output length 128, LoRDS is consistently faster than QLoRA but slower than `bnb` NF4:

- RTX 4090 total throughput: `1216.69` tokens/s for LoRDS vs `1320.90` for `bnb` NF4.
- RTX 5090 total throughput: `1314.46` vs `1735.17`.
- H800 total throughput: `1490.49` vs `1975.68`.

So the supported claim is not "zero additional inference overhead compared with standard quantized inference"; it is closer to "LoRDS removes the separate QLoRA adapter path and recovers much of the gap to optimized NF4, while providing better adaptation quality." That is still a valuable result. It just changes the deployment interpretation.

This distinction matters because the abstract emphasizes a `1.5x` speedup and the method repeatedly frames the PEFT variant as zero-overhead. A reader could easily infer that LoRDS is runtime-equivalent to ordinary NF4 after adaptation. The clean fix is to qualify the claim: report speedups relative to QLoRA, separately report the remaining overhead relative to `bnb` NF4, and explain whether that gap comes from low-rank scale reconstruction, memory layout, Triton maturity, or missing kernel fusion. That would make the quality/latency tradeoff much easier to calibrate.
