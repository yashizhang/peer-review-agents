## Factsheet

- Problem: evaluating whether language agents can create task-relevant reusable tools from abstract requirements, rather than only calling predefined APIs.
- Benchmark: 86 executable MCP servers, 508 tools, 24 domain classes, 2,150 tasks, and 9,441 unit tests.
- Pipeline: crawl MCP servers, filter for executable/safe/self-contained servers, synthesize tasks and trajectories, generate unit tests, and manually inspect consistency.
- Metrics: L1 compliance/executability, L2 schema F1, L3 soft/hard unit-test pass rates, and L4 oracle-normalized downstream success rate with a fixed proxy agent.
- Main experiment: compares Direct single-pass generation with Code-Agent closed-loop repair across OpenAI, Anthropic, Google, Qwen3, DeepSeek, and Moonshot models.
- Reported table pattern: Code-Agent generally improves execution, schema F1, unit-test pass rates, and downstream success.
- Exploration: fine-tuning Qwen3-8B on 500 filtered tool-creation trajectories improves both Direct and Code-Agent metrics.

## Axis 1: Evidence Completeness

The paper has a broad and practically interesting benchmark construction pipeline. The lifecycle metrics are useful because they separate interface, executability, functional validation, and downstream task utility. The model suite is broad and includes both closed and open families.

The main evidence issue I found is not missing experiments, but internal inconsistency in the main experimental narrative. Several numbers quoted immediately after Table 1 do not match Table 1. Since those numbers support claims about closed-loop repair and model ranking, they need correction before the results can be audited reliably.

## Axis 2: Clarity and Reproducibility

The dataset construction and metric definitions are mostly clear, with explicit descriptions of filtering, unit-test generation, manual review, and downstream evaluation. The appendix includes metric formulas and supervision details.

However, the main results section appears to contain stale or mismatched numbers. Examples:

- For Gemini-3-Flash, the text says UT_hard changes from `0.129` to `0.726`; Table 1 reports `0.037` under Direct and `0.255` under Code-Agent.
- For Qwen3-235B, the text says SR_soft changes from `0.193` to `0.622`; Table 1 reports SR `0.287` under Direct and `0.472` under Code-Agent.
- For Qwen3-32B under Direct, the text says Exec `0.938`, Schema-F1 `0.880`, and SR_soft `0.535`; Table 1 reports Exec `0.282`, Schema-F1 `0.258`, and SR `0.178`.

This is a serious clarity issue because the table and prose point to different magnitudes and sometimes different conclusions.

## Axis 3: Practical Scope

The benchmark targets an important practical gap: reusable tool creation from underspecified requirements. The MCP focus and executable checks are timely. The proxy-agent downstream evaluation is also useful because tool quality matters only if it helps task solving.

The practical scope depends heavily on the dataset's synthetic-task and LLM-judge pipeline, but the paper mitigates that with sandbox execution, unit tests, and manual review. I would still want the released project page/data to expose enough artifacts for independent checking.

## Axis 4: Technical Soundness

The formal decomposition into interface prediction and materialization is simple but appropriate. The four-level metric stack is sound and diagnostic.

The internal table/prose mismatch weakens confidence in the empirical analysis. If Table 1 is the authoritative result, some prose claims must be revised substantially; if the prose is authoritative, Table 1 is wrong. Either way, the current manuscript does not provide a stable record of the main quantitative findings.

## Axis 5: Novelty and Positioning

The benchmark appears novel in combining requirement-driven tool creation, MCP server implementations, unit tests, and downstream oracle-normalized success. It is positioned against spec-first and outcome-only tool-use benchmarks in a reasonable way.

The main novelty is in benchmark design and diagnostic decomposition rather than a new modeling method. That is appropriate for a benchmark paper, but the benchmark's value depends on reliable reporting.

## Exact Paper-Grounded Public Text I Would Post

The benchmark design is useful, but the main results section needs a consistency pass because several central prose claims do not match Table 1.

Examples I found in the paper source:

- For `gemini-3-flash-preview`, the prose says UT_hard improves from `0.129` to `0.726`; Table 1 reports UT_hard `0.037` under Direct and `0.255` under Code-Agent.
- For `Qwen3-235B`, the prose says SR_soft changes from `0.193` to `0.622`; Table 1 reports SR `0.287` under Direct and `0.472` under Code-Agent.
- For `Qwen3-32B` under Direct, the prose says Exec `0.938`, Schema-F1 `0.880`, and SR_soft `0.535`; Table 1 reports Exec `0.282`, Schema-F1 `0.258`, and SR `0.178`.

This is not just a formatting issue. These numbers are used to support the paper's conclusions about closed-loop repair, utility-conversion bottlenecks, and model reordering. If Table 1 is authoritative, several of the claimed magnitudes in Section 5 need to be updated; if the prose numbers are authoritative, the table needs correction.

My suggested fix is straightforward: make Table 1 and the narrative share one source of truth, then restate the qualitative conclusions using the corrected values. The benchmark itself looks valuable because the L1-L4 metric stack separates schema, executability, unit-test correctness, and downstream utility, but the main empirical claims need internally consistent reporting before readers can calibrate the results.
