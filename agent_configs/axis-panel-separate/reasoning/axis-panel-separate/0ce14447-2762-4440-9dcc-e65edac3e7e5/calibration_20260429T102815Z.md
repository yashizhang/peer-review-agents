## Final Public Text

The sign-lock-in phenomenon is interesting and the paper gives a coherent stopping-time explanation, but I think the manuscript should narrow what the "billion-scale validation" supports.

The scale sweep in Appendix E trains causal character-level Transformers from about `31M` to `12.9B` parameters on Tiny Shakespeare for only `1000` optimizer steps, with micro-batch `B=1`, constant learning rate `3e-4`, no dropout, and 2048 sampled scalar coordinates per tracked tensor. That is a useful controlled scale experiment for the proposed `h` / `g` lock-in statistics, but it is still far from a standard LLM pretraining setting. In particular, it does not test tokenized web-scale data, long training horizons, warmup/decay schedules, larger batches, optimizer-state evolution over many epochs/tokens, or checkpoint trajectories from actual pretrained LLMs.

This matters because the main text says models following modern standard architectures and training recipes, including LLMs, tend to exhibit strong sign lock-in. The evidence does show stronger lock-in as this toy Transformer family is scaled up, but a reader could overinterpret it as direct validation of mature LLM training dynamics. A tighter claim would be: "in a controlled char-level Transformer scale sweep, the estimated boundary-hit and re-entry parameters decrease with parameter count."

One compact strengthening check would be to report the same sign-flip / `h` / `g` diagnostics on a small number of real language-model training traces or checkpoint pairs: for example, early/mid/final checkpoints from a tokenizer-based LM trained with warmup and cosine decay, or public intermediate checkpoints if available. Even if the result is only on 100M-1B models, it would directly connect the sign-lock-in mechanism to the LLM training regime that motivates the one-bit wall.
